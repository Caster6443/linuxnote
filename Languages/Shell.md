
在真正开始编写代码之前，我们需要先打破 Windows 留给我们的思维惯性，重新认识 Linux 对待文件的哲学。

**一. 命名的潜规则**

为了统一规范，我们通常会用 **`.sh`** 作为 Shell 脚本的后缀。但这**仅仅是为了方便“人类”识别和编辑器的高亮显示**。 在 Linux 的内核眼中，根本没有“后缀名决定文件属性”这种机制。无论你叫 `script.sh` 还是 `script.txt`，Linux 只看重两点：文件里面写了什么代码，以及你给了它什么权限。

**二. 脚本运行的一些机制**

既然不靠双击后缀运行，那写好的文本是怎么变成“可执行程序”的？我们有三种调用方式：

- **1：标准运行方式** 说到底，脚本就是一个需要被执行的文件。你必须先赋予它“可执行权限”：`chmod +x xxx.sh`。 然后，你需要明确告诉系统它的位置才能运行（哪怕它就在当前目录）：`./xxx.sh`。
    
- **2：偷懒运行（调用shell解释器）** 如果是写一些用完即弃的临时脚本，其实可以省去修改权限的步骤。 直接把脚本文件当作参数，喂给解释器去读：`bash xxx.sh`。系统会直接运行它，完全不看它有没有可执行权限。
    
- **3：化作系统原生命令** 如果你的脚本极其好用，你想让它像 `ls` 或 `cd` 一样随叫随到怎么办？ 只需要把它移动到系统的 **`$PATH` 环境变量** 所包含的任何一个目录（比如 `/usr/local/bin`）中。从此以后，你在任何目录下，直接输入 `xxx.sh`（甚至可以把后缀去掉直接输入 `xxx`），它就能直接运行！

## 基础语法

### 1. 读取用户输入

```shell
#!/bin/bash
echo "请输入名字: "
read user_name
echo "欢迎, $user_name"
```

### 2. if..else判断语句

```shell
#!/bin/bash
score=85
if [ $score -ge 90 ]; then
  echo "nice~"
elif [ $score -ge 60 ]; then
  echo "good"
else
  echo "well"
fi

```

### 3. for/while循环语句

```shell
#!/bin/bash
for i in {1..10}; do
  echo "$i"
done
```

```shell
#!/bin/bash
count=3
while [ $count -ge 1 ]; do
  echo "倒数: $count"
  ((count--))
done
```

### 4.接收命令行参数

在脚本内部，系统准备了一套专门的**内置变量**来接收这些参数：

- **`$0`**：脚本本身的名字（比如 `./backup.sh`）。
    
- **`$1`, `$2`, `$3`...**：传入的第 1 个、第 2 个、第 3 个参数。
    
- **`$#`**：传入参数的**总个数**。这个极其有用，常用来做安全检查（防呆设计）。
    
- **`$@`**：代表所有的参数列表。常用于 `for` 循环把参数挨个遍历一遍。

写一个简单的“打包备份”脚本 (`backup.sh`)

```shell
#!/bin/bash
if [[ $# -ne 1 ]]; then
  echo "非法输入! 示例用法: $0 </path/to/dir/>"
  exit 1
fi

target_dir=$1

if [[ ! -d "$target_dir" ]]; then
  echo "error, target not found!"
  exit 2
fi

echo "backuping ..."
tar -czf "$1.tar.gz" "$1"
echo "done"

```

这里可能有几点疑惑，比如为什么要在if的执行语句中写入exit，而不是通过判断语句的返回值来判断，比如输入参数大于1，判断语句返回了非零值表示假，为什么还要多此一举在if执行语句里写exit呢？这是因为返回值被if吃掉了

在 Shell 里，除非你使用了特殊的严格模式（`set -e`），否则**一条命令哪怕执行失败（返回非 0），脚本也会硬着头皮往下执行下一行。**

### 5.算术运算

Shell 极其死板，它**默认把所有东西都当成纯文本（字符串）**。 如果你在脚本里写 `a=1+1`，然后 `echo $a`，屏幕上会原封不动地打印出 `1+1`，而不是 `2`

对此，我们只需要使用**双小括号 `$(( 算式 ))`**，把算式塞进去，shell就会有计算功能

**双小括号里的三大“特权”：**

1. **空格自由：** 里面怎么加空格都不会报错（`$((1+1))` 和 `$(( 1 + 1 ))` 都行）。
    
2. **变量免 `$` ：** 在双小括号里面用到变量时，前面的 `$` 可以省略，系统能自动认出来。
    
3. **只限整数：** Shell 原生只懂整数运算。`10 / 3` 的结果是 `3`，小数部分会被直接抹掉。

写个脚本做演示

```shell
#!/bin/bash
x=10
y=3

# 常规运算
sum=$((x + y))
div=$((x / y))
mod=$((x % y))

# 自增操作
((x++))

echo "10 除以 3 的商是 $div，余数是 $mod"
```

### 6.命令替换

你在终端敲 `date`，屏幕会显示当前时间。敲 `whoami`，会显示当前登录的用户名。但写脚本时，我不想让它打印在屏幕上，我**想把这些结果存进变量里**，留着后面拼接文件名或者写日志用。
这时，就需要引入命令替换的机制: 把你要执行的命令放进 `$()` 里，Shell 会悄悄在后台执行它，然后把原本要喷到屏幕上的输出“截胡”，变成一个字符串交给你。
这个在linux基础章节里面说过，有两种命令替换方式，反引号\`命令\` 和$(命令)，前者兼容度更高，但这是比较老的写法了

依旧写个脚本测试

```shell
#!/bin/bash

# 获取 date 命令的输出，按照 "年-月-日" 格式化
current_time=$(date "+%Y-%m-%d")

# 截获 ls | wc -l 的输出（统计当前目录下有多少个文件）
file_count=$(ls | wc -l)

# 拿截获的数据来拼接输出
echo "今天是 $current_time，当前目录下共有 $file_count 个文件。"

# 还可以用变量名来创建带日期的文件！
touch "backup_$current_time.log"
```

### 7.函数

当你的脚本越写越长，如果有一段检查网络的逻辑你需要用到 5 次，总不能复制粘贴 5 遍吧？你需要用“函数”把代码打包。

**Shell 函数与 Python/C/Java 等高级语言的函数截然不同，有 4 个极其反直觉的坑！**

1. **定义时括号必须是空的：** `函数名() { ... }`，不管你要传几个参数，括号里**绝对不写东西**。
    
2. **调用时没有括号：** 调用函数就像执行普通的 Linux 命令一样，直接 `函数名 参数1 参数2`，用空格隔开。
    
3. **函数内部的 `$1` 是专属的：** 在函数里面写 `$1`，代表的是**传给这个函数的**第一个参数。它会临时覆盖掉传给整个脚本的那个 `$1`！
    
4. **无 return 机制：** Shell 的 `return` 极其寒酸，它**只能返回 0-255 的状态码**（代表成功还是失败，供 `if` 判断使用）。如果你想让函数返回一个计算出来的字符串，必须在函数里用 `echo` 打印，然后在外面用上面的命令替换机制`$()` 把打印的内容截获下来！

脚本测试

```shell
#!/bin/bash

# 定义函数 (括号必须为空)
say_hello() {
  # 建议：在函数内部定义的变量加上 local 关键字，防止污染外面的同名变量
  local user_name=$1
  local age=$2

  echo "你好 $user_name, 你 $age 岁了吗?"
}

# 调用函数 (不加括号，像敲命令一样传参)
say_hello "Alice" 25
say_hello "Gugugaga" 30

# 进阶演示：如何让函数“返回”一个处理过的字符串？
get_system_status() {
  # 不用return, 而是用echo输出结果
  echo "正常运行中"
}

# 在外面用 $() 把 echo 喷出的结果截获，存进 status 变量
status=$(get_system_status)
echo "当前服务器状态: $status"
```

为什么要用local?

因为**在 Shell 里（巨大的坑）：** 在函数里定义的任何变量，**默认全都是全局变量！** 只要函数被执行过一次，里面的变量就会“泄漏”到整个脚本中，无情地覆盖掉你在外面定义的同名变量。



### Shell 编程避坑与机制

#### 一、 Shell 语法的“反直觉”规则

初学 Shell 最容易报语法错误，往往是因为没有理解以下几条底层规则：

1. **变量赋值，绝对不能有空格**
    
    - **规范：** 必须写成 `score=85`。
        
    - **雷区：** 一旦有空格（如 `score = 85`），Shell 就会把 `score` 误认作一个要执行的命令，把 `=` 当作它的参数，从而引发 `command not found` 报错。
        
2. **左中括号 `[` 根本不是标点，它是一个“程序”**
    
    - **真相：** `[` 是早期 UNIX 系统中 `test` 命令的“马甲”（系统里真实存在一个名为 `[` 的可执行文件）。如果你使用 `which [` 命令，是可以查到它的具体路径的。那么就引入了一个重点，其实if等语句后面跟着的中括号从来就不是什么判断体，而是一个命令，它们看的是这个命令的退出状态码是0(成功)还是非0(失败)，也就是说，我可以写 `if [...]`,我也可以写 `if id nginx`, 在这个写法中，if实际上就是在看`id nginx`这个命令的退出状态码
        
3. **中括号的“空格”与“闭合”规则**
    
    - **空格规则：** 既然 `[` 是命令，那么 `-ge`、`5`、`3` 都是传给它的参数。命令与参数之间**必须用空格隔开**，例如：`[ 5 -ge 3 ]`。
        
    - **闭合规则：** `[` 命令内部有一条硬性规定，传入的最后一个参数必须是 `]`，这仅仅是为了让代码在视觉上显得闭合。

#### 二、 常用判断条件速查字典

##### 1. 数字比较符（首字母记忆法）

Shell 中的数字比较符看起来毫无规律，但其实全部是英文单词首字母的缩写，无需死记硬背：

- `-eq` = **e**qual (等于)
    
- `-ne` = **n**ot **e**qual (不等于)
    
- `-gt` = **g**reater **t**han (大于)
    
- `-lt` = **l**ess **t**han (小于)
    
- `-ge` = **g**reater or **e**qual (大于等于)
    
- `-le` = **l**ess or **e**qual (小于等于)
    

##### 2. 文件系统探测（给路径“查户口”）

- `-e` (Exist)：路径**是否存在**（不管是文件、目录还是快捷方式，在就行）。 例：`[[ -e "/etc/passwd" ]]`
    
- `-f` (File)：必须存在，且必须是一个普通的**文本/数据文件**（不能是目录）。 例：`[[ -f "data.txt" ]]`
    
- `-d` (Directory)：必须存在，且必须是一个**目录（文件夹）**。 例：`[[ -d "/var/log" ]]`
    
- `-s` (Size)：文件存在，并且**大小不为 0**（里面有内容，常用于检查日志）。 例：`[[ -s "error.log" ]]`
    

##### 3. 字符串状态探测（检查变量是否为空）

- `-z` (Zero)：判断字符串长度**是否为 0**（空值）。防呆设计必备，没输入名字则为真。 例：`[[ -z "$name" ]]`
    
- `-n` (Non-zero)：判断字符串长度**是否非空**（有内容）。输入了名字则为真。 例：`[[ -n "$name" ]]`

#### 三、 核心机制剖析：单中括号 `[ ]` 的两大雷区

在脚本执行判断语句前，Shell 会先执行一个核心预处理机制：**Shell 扩展（Shell Expansion）和单词拆分（Word Splitting）**。它的作用是先把变量替换为具体的值，再进行切割。这个机制导致了 `[ ]` 极易翻车。

##### 雷区 1：变量为空值时的“语法残缺”

假设我们有以下判断：

```Bash
if [ $name == "qq" ]; then ...
```

如果 `$name` 为空，Shell 预处理后会直接把 `$name` 擦除，代码变成：

```Bash
if [ == "qq" ]; then ...
```

**结果：** 语法残缺，缺少左边的比较对象，直接报错。

##### 雷区 2：变量值包含空格时的“参数过多”

假设用户输入了带空格的名字：`name="Genshin Impact"`。再让代码走一遍流水线：

1. **预处理替换：** 代码变成了 `[ Genshin Impact == "qq" ]`。
    
2. **致命的切词：** 因为 `Genshin` 和 `Impact` 中间有空格，Shell 会把它们切成两个独立的词！切出来的参数变成了：`Genshin`、`Impact`、`==`、`qq`。
    
3. **执行并崩溃：** `[` 命令原本期待 `[ 左边参数 比较符 右边参数 ]`。结果现在左边塞了两个参数给它，它当场罢工，抛出经典报错：`bash: [: too many arguments`（参数过多）。


如果使用 `[[ $name == "qq" ]]`，就完全不会出现上述问题：

- `[[ ]]` 是 Bash 的内部关键字，它在内部**禁用了单词拆分**机制。
    
- 即使变量为空，或者包含空格，它都会安全地将整个变量值视为一个被保护的独立整体（效果等同于隐式加上了双引号）。
    

**最佳实践：只要在 Bash 环境下，判断语句无脑使用 `[[ ]]` 即可。**
